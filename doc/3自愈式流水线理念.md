# DevOpsAgent 自愈式流水线设计文档

## 版本信息

- **版本**: v4.0.0 (Agent Skill 架构版)
- **更新日期**: 2026-05-06
- **部署方式**: Agent Skill 集成 (不再需要独立 Bridge 服务)

---

## 一、项目概述

### 1.1 核心目标

DevOpsAgent 是一个高度自治的 CI/CD 自愈系统，通过 AI 驱动的自动化流程，将传统的单向 Jenkins 流水线转变为智能闭环修复系统。

**核心价值主张**:

- **自动化闭环**: 构建失败 → AI 诊断 → 生成修复 → 自动验证
- **高度自治**: 无需人工干预，全自动运行
- **工程化管理**: 所有 Job 定义通过 JJB (Jenkins Job Builder) YAML 配置管理
- **Skill 集成**: 作为 Agent Skill 原生集成，无需独立中间件服务

### 1.2 架构演进

| 版本 | 架构 | 说明 |
|------|------|------|
| **v1.0.0** | Bridge 服务 + 创建新 Job | 独立 HTTP 服务，每次修复创建新 Job |
| **v2.0.0** | Bridge 服务 + JJB 更新原 Job | 独立 HTTP 服务，更新原 Job 配置 |
| **v3.0.0** | Docker Compose + Bridge 服务 | 容器化编排，包含 PostgreSQL、Redis、GitLab |
| **v4.0.0** | **Agent Skill 架构** | **集成到 Agent/Trae，无独立服务** |

### 1.3 关于 PostgreSQL 和 Redis

**重要说明 (v4.0.0 更新)**:

**自愈功能本身不需要 PostgreSQL 或 Redis**：
- 自愈状态存储在本地 JSON 文件 (`.self-heal-state.json`)
- 配置管理通过本地 YAML 文件 (JJB 配置)
- AI 调用通过 Agent CLI

**关于 GitLab**:

在 v4.0.0 中，GitLab CE 使用**内置的 PostgreSQL 和 Redis**，无需单独安装这些服务：

```yaml
# docker-compose.yml 中的配置 (v4.0.0)
gitlab:
  image: gitlab/gitlab-ce:latest
  environment:
    - GITLAB_OMNIBUS_CONFIG="
        external_url 'http://${GITLAB_HOSTNAME:-gitlab.devopsagent.local}';
        gitlab_rails['gitlab_shell_ssh_port'] = 2222;
      "
  # 注意: 没有设置 postgresql['enable'] = false
  # 注意: 没有设置 redis['enable'] = false
  # 这意味着 GitLab 使用内置的 PostgreSQL 和 Redis
```

**架构对比**:

| 版本 | PostgreSQL | Redis | 说明 |
|------|-----------|-------|------|
| **v3.0.0** | 独立 Docker 容器 | 独立 Docker 容器 | GitLab 禁用内置数据库，使用外部服务 |
| **v4.0.0** | **GitLab 内置** | **GitLab 内置** | GitLab CE 自带，无需单独管理 |

**服务清单 (v4.0.0)**:

| 服务 | 是否必需 | 说明 | 是否独立服务 |
|------|---------|------|-------------|
| **Jenkins** | ✅ 必需 | CI/CD 引擎 | ✅ Docker 容器 |
| **Agent** | ✅ 必需 | AI 网关 + Skill 引擎 | ✅ Docker 容器 |
| **GitLab CE** | ❌ 可选 | 代码仓库 (使用内置 PostgreSQL/Redis) | ✅ Docker 容器 (可选) |
| **PostgreSQL** | ❌ 已移除 | **不再需要单独安装** | ❌ 已移除 |
| **Redis** | ❌ 已移除 | **不再需要单独安装** | ❌ 已移除 |
| **Bridge** | ❌ 已移除 | **整合为 Agent Skill** | ❌ 已移除 |

| 维度 | v3.0.0 (Docker Compose + Bridge) | v4.0.0 (Agent Skill) |

| **与 AI 集成** | 外部调用 Agent | **内部集成，无缝协作** |

## 三、Skill 结构与使用

### 3.1 Skill 文件结构

Skill 存储在 `.trae/skills/ci-selfheal/` 目录：

```
.trae/skills/ci-selfheal/
├── SKILL.md              # Skill 定义 (必需)
│                         # 包含:
│                         # - name: "ci-selfheal"
│                         # - description: 何时激活此 Skill
│                         # - 详细使用指南
│
├── ci_selfheal.py        # 核心自愈逻辑
│                         # 主入口: process_event(event)
│
├── jenkins_client.py     # Jenkins API 封装
│                         # - 拉取构建日志
│                         # - 触发构建
│                         # - 获取 Job 配置
│
├── jjb_manager.py        # JJB 配置管理
│                         # - 查找 YAML 配置
│                         # - 读取/更新 dsl
│                         # - 执行 jenkins-jobs 命令
│
└── webhook_listener.py   # 可选: 极简 Webhook 接收器
                          # - 监听端口 5000
                          # - 接收 Jenkins Webhook
                          # - 调用 process_event()
```

### 3.2 Skill 触发条件

**当以下情况发生时，此 Skill 会被激活**:

1. **Jenkins 构建失败事件**:
   - 收到包含 `"status": "FAILURE"` 的 Webhook
   - 用户提供了构建失败的 Job 名称和构建号

2. **用户询问 CI 相关问题**:
   - "如何修复这个 Jenkins 构建失败?"
   - "这个 Pipeline 为什么失败了?"
   - "帮我看一下 CI 错误"

3. **关键词触发**:
   - Jenkins、Pipeline、构建失败、CI/CD 错误
   - JJB、Jenkins Job Builder
   - 自愈、自动修复

### 3.3 使用方法

#### 方法 1: 作为 Skill 自动激活

当 Skill 被正确配置后，Agent/Trae 会在以下情况自动使用此 Skill:

- 检测到 Jenkins 构建失败事件
- 用户询问与 CI 修复相关的问题

#### 方法 2: Python 模块调用

```python
from .trae.skills.ci_selfheal.ci_selfheal import process_event

# 构建事件
event = {
    "jobName": "example-pipeline",
    "buildNumber": 42,
    "status": "FAILURE"
}

# 处理事件
result = process_event(event)
print(result)
```

#### 方法 3: 命令行调用

```bash
# 处理单个失败事件
python .trae/skills/ci-selfheal/ci_selfheal.py \
  --job-name "example-pipeline" \
  --build-number 42 \
  --status FAILURE

# 从 JSON 文件读取事件
python .trae/skills/ci-selfheal/ci_selfheal.py \
  --event-file event.json

# 查看当前状态
python .trae/skills/ci-selfheal/ci_selfheal.py --status
```

#### 方法 4: Webhook 接收器 (用于 Jenkins 集成)

```bash
# 启动 Webhook 服务
python .trae/skills/ci-selfheal/webhook_listener.py --port 5000
```

然后在 Jenkins Pipeline 中配置 Webhook:

```groovy
pipeline {
    agent any
    
    stages {
        stage('Build') {
            steps {
                sh 'make build'
            }
        }
    }
    
    post {
        always {
            script {
                try {
                    def payload = [
                        jobName: env.JOB_NAME,
                        buildNumber: env.BUILD_NUMBER,
                        status: currentBuild.currentResult
                    ]
                    
                    sh """
                        curl -s -X POST \\
                          -H "Content-Type: application/json" \\
                          -d '${groovy.json.JsonOutput.toJson(payload)}' \\
                          http://127.0.0.1:5000/webhook/jenkins
                    """
                } catch (Exception e) {
                    echo "Failed to notify: ${e.getMessage()}"
                }
            }
        }
    }
}
```

### 4.2 最大重试机制

系统设计了最多 5 轮的修复重试机制：

```
构建 #1 (失败)
    │
    ▼
第 1 轮修复 → 更新 JJB 配置 → 构建 #2
    │
    ├── 成功? ──► 是 ──► 结束 (自愈成功)
    │
    └── 失败? ──► 第 2 轮修复 → 构建 #3
                      │
                      ├── 成功? ──► 结束
                      │
                      └── 失败? ──► 第 3 轮 → ...
                                        │
                                        └── 第 5 轮后仍失败
                                                │
                                                ▼
                                           停止自愈
                                           记录 max_retry 状态
                                           等待人工介入
```

**重试终止条件**:

1. **构建成功** → 终止，记录 `status: "success"`
2. **达到最大重试次数** (默认 5) → 终止，记录 `status: "max_retry"`
3. **AI 返回 `CANNOT_FIX`** → 终止，记录 `status: "failed"`
4. **连续相同错误 3 次** → 终止，防止无限循环

    token="your_token"
)

# 测试连接
try:
    config = jenkins.get_job_config("your-job-name")
    print("连接成功!")
except Exception as e:
    print(f"连接失败: {e}")
```

### 8.3 无法找到 JJB 配置文件

**可能原因**:
1. `JJB_CONFIG_PATH` 配置错误
2. Job 名称与 YAML 文件名不匹配
3. YAML 文件中 `job.name` 字段不正确

**排查方法**:
```python
from jjb_manager import JJBManager

jjb = JJBManager(config_path="./jjb-configs")

# 列出所有配置的 Job
jobs = jjb.list_all_jobs()
print(f"已配置的 Job: {jobs}")

# 查找特定 Job
yaml_file = jjb.find_job_yaml("your-job-name")
print(f"YAML 文件: {yaml_file}")
```

### 8.4 AI 调用失败

**可能原因**:
1. Agent 容器未运行
2. 模型名称错误
3. API 凭证问题

**排查方法**:
```bash
# 检查 Agent 容器状态
docker ps | grep agent

docker exec agent node agent.mjs infer model run \
