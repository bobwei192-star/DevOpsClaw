## 第一章 · 背景与目标

### 1.1 现状痛点

| 痛点维度 | 具体表现 |
|---------|---------|
| CI 失败频率 | 日均构建失败次数高，部分团队失败率超过 15% |
| 人工介入耗时 | 从失败发现到修复平均耗时 2~4 小时，夜间/周末无人响应时延长至 8 小时以上 |
| 人力覆盖不足 | 非工作时段（夜间、周末、节假日）缺乏运维与开发人员响应，失败构建长时间挂起 |
| 重复性问题 | 编译错误、依赖冲突、配置语法错误等类型占比超过 60%，但每次都需人工排查 |
| 上下文缺失 | 开发人员接手时往往需要重新收集日志、环境信息、最近变更，诊断成本高 |

### 1.2 核心目标

| 指标 | 目标值 | 说明 |
|------|--------|------|
| MTTR（平均修复时间） | 降低 70% 以上 | 从小时级降至分钟级 |
| 自动修复成功率 | ≥ 60%（可自愈类型中） | 编译/依赖/配置类错误 |
| 人工介入率 | ≤ 30% | 仅业务逻辑 Bug、安全阻断等需人工 |
| 误修率 | ≤ 5% | 自动修复后引入新问题或修复错误的比率 |
| 夜间/周末覆盖 | 7×24 小时自动响应 | 无需人工值守 |

### 1.3 适用范围

**纳入自愈范围的 Repo**：
- 已加入白名单的仓库（由运维团队审核准入）
- 非保护分支（如 `fix/*`、`feat/*`、`dev/*`）
- 构建失败类型为编译错误、依赖冲突、配置语法错误、环境变量缺失、测试脚本错误

**暂不纳入的情况**：
- 主干分支（`main` / `master` / `release/*`）
- 安全门禁阻断（如 SonarQube 质量门禁、漏洞扫描失败）
- 业务逻辑断言失败（功能回归、需求变更导致的测试失败）
- 基础设施全局故障（Jenkins 宕机、网络分区、第三方服务不可用）

### 1.4 不做的事（明确边界）

| 边界项 | 说明 |
|--------|------|
| 不碰主干 | 任何自动修复不直接提交到 `main` / `master` / `release/*` |
| 不修业务逻辑 | 不修改 `src/main/` 下的业务源代码，仅修复构建配置、测试脚本、Pipeline DSL |
| 不绕过安全门禁 | 安全扫描、合规检查失败的构建不进入自愈流程 |
| 不自动合并 | 自动修复后必须经人工 Review 才能合并 |
| 不替代人工决策 | AI 仅提供修复建议，最终决策权在开发/运维团队 |

### 1.5 成功度量

| 度量指标 | 采集方式 | 目标 |
|---------|---------|------|
| 自愈成功率 | 自动修复后 CI 重建通过的次数 / 总尝试次数 | ≥ 60% |
| PR 接受率 | 自动提 PR 后被合并或接受的比率 | ≥ 80% |
| 误修率 | 自动修复后引入新失败或错误修复的比率 | ≤ 5% |
| 熔断触发频率 | 单位时间内触发熔断的次数 | 逐月下降 |
| 平均修复轮次 | 从首次失败到最终修复成功的平均重试次数 | ≤ 2 次 |

### 2.2 五阶段流水线

| 阶段 | 名称 | 职责 | 对应模块 |
|------|------|------|---------|
| 阶段 0 | 触发门控 (G0) | 白名单校验、分支保护、去重锁 | `gate/` |
| 阶段 1 | 信息收集 (S1) | 日志、DSL、diff、环境快照采集 | `collector/` |
| 阶段 2 | 故障预判 (G1) | 拦截不可自愈场景，分类错误类型 | `gate/precheck` |
| 阶段 3 | AI 分析 (S2) | 根因定位、修复生成、置信度评分 | `diagnoser/` |
| 阶段 4 | 修复决策 (S3) | 决策矩阵分流、分支/Patch/报告生成 | `decider/` |
| 阶段 5 | 验证闭环 (S4) | 触发重建、PR 创建、结果反馈 | `verifier/` |

### 2.3 五大决策规则

| 规则编号 | 规则名称 | 触发条件 | 处理动作 |
|---------|---------|---------|---------|
| G0 | 白名单门控 | 非授权 Repo / 保护分支 / 重复事件 | 拦截，记录日志，通知运维 |
| G1 | 故障预判 | Flaky Test / 基建宕机 / 安全阻断 / 业务逻辑 Bug | 拦截，生成诊断报告，通知人工 |
| G2 | 置信度分级 | AI 输出置信度 < 0.60 | 降级为仅诊断报告，不生成修复 |
| G3 | 权限判断 | 无 Git 写权限 / 无 Git 仓库 | 生成 Patch 文件或诊断报告，不自动 Push |
| G4 | 熔断保护 | 连续重试超过 5 次或时间窗口内全局失败率过高 | 停止自动修复，分级告警，人工兜底 |

### 2.4 终结状态

| 状态 | 说明 | 后续动作 |
|------|------|---------|
| 自愈成功 | 自动修复后 CI 重建通过，PR 已创建 | 等待人工 Review，结果入库 |
| 被拒 | PR 被人工拒绝，或 Patch 被判定不可行 | 记录拒绝原因，更新样本库 |
| 门控拦截 | G0/G1 规则触发，未进入 AI 诊断 | 通知相关方，记录审计日志 |
| 熔断 | 重试超限或全局熔断触发 | 停止自愈，分级告警，人工介入 |
| 仅诊断 | 置信度低或权限不足，仅输出诊断报告 | 供人工参考，不自动修复 |

### 2.5 技术选型

| 模块 | 技术栈 | 说明 |
|------|--------|------|
| AI 基座模型 | GPT-4 / Claude / DeepSeek / 本地模型 | 支持多模型切换，默认 DeepSeek |
| CI 平台 | Jenkins（主）+ GitLab CI / GitHub Actions（扩展） | 通过 REST API 交互 |
| 消息队列 | 内置 HTTP Webhook + 可选 Redis/RabbitMQ | 轻量级场景直接 Webhook |
| 存储 | 本地 JSON 状态文件 + 可选 PostgreSQL/MongoDB | 状态、历史、样本库 |
| 并发锁 | 文件锁 / Redis 分布式锁 | 防止同一事件重复处理 |
| 通知渠道 | Slack / 钉钉 / 邮件 / n8n 工作流 | 可配置多通道 |

## 第三章 · 安全门控设计

### 3.1 Repo 白名单机制

| 维度 | 设计方案 |
|------|---------|
| 配置方式 | 支持 YAML/JSON 配置文件或数据库表存储白名单 |
| 白名单内容 | Repo 名称、允许自愈的分支前缀、最大重试次数、通知对象 |
| 动态更新 | 支持热更新（文件变更自动加载）或 API 动态增删 |
| 准入审核 | 新 Repo 加入需经运维团队审核，确认构建类型适合自愈 |

### 3.2 分支保护规则

| 分支类型 | 处理策略 |
|---------|---------|
| 主干分支 | `main` / `master` / `release/*` → 直接拦截，不进入自愈 |
| 保护分支 | 受 Git 分支保护规则限制的分支 → 拦截 |
| 允许分支 | `fix/*` / `feat/*` / `dev/*` / `ci-fix/*` → 允许进入 |
| 分支命名规范 | 自动修复创建的分支统一以 `fix/ci-selfheal-{job}-{build}` 命名 |

### 3.3 去重锁设计

| 维度 | 设计方案 |
|------|---------|
| 锁粒度 | `repo + 分支 + 错误类型` 三元组 |
| 锁实现 | 文件锁（轻量）或 Redis 分布式锁（集群） |
| 锁过期策略 | 默认 30 分钟自动释放，防止死锁 |
| 锁冲突处理 | 同一三元组在锁有效期内的新事件直接丢弃，记录日志 |

### 3.4 故障预判过滤器

| 不可自愈类型 | 识别特征 | 处理动作 |
|-------------|---------|---------|
| Flaky Test | 同一测试在不同时段/环境下结果不一致，历史记录中波动率高 | 拦截，标记为不稳定测试，通知开发 |
| 基础设施故障 | Jenkins Slave 离线、磁盘满、网络不可达、DNS 解析失败 | 拦截，通知运维团队 |
| 安全门禁阻断 | SonarQube 质量门禁、漏洞扫描、合规检查失败 | 拦截，通知安全团队 |
| 第三方服务不可用 | 外部 API 超时、依赖服务返回 5xx | 拦截，通知运维 |
| 业务逻辑 Bug | 断言失败指向 `src/main/` 代码变更，功能回归 | 拦截，生成诊断报告，通知开发 |

### 3.5 拦截后的行为

| 维度 | 设计方案 |
|------|---------|
| 通知模板 | 按拦截原因提供标准化通知模板（含构建信息、拦截原因、建议动作） |
| 日志记录 | 全链路审计日志，记录拦截时间、规则、触发事件 |
| 审计追踪 | 所有拦截事件持久化存储，支持按 Repo/分支/时间范围查询 |

## 第四章 · 信息收集层

### 4.1 数据采集清单

| 数据类型 | 内容说明 | 用途 |
|---------|---------|------|
| 构建日志 (raw) | 完整 Console Output，含时间戳 | AI 诊断主输入 |
| 构建日志 (parsed) | 按阶段拆分（Checkout/Build/Test/Deploy） | 快速定位失败阶段 |
| Pipeline DSL | Jenkinsfile / JJB YAML / GitLab CI YAML | 理解构建定义，生成修复 |
| 源码 diff | 触发构建的 Commit 与上一次成功构建的 diff | 定位变更引入的问题 |
| 环境快照 | 构建节点信息、环境变量、工具版本、依赖清单 | 排查环境类问题 |
| 历史构建记录 | 最近 N 次构建结果、失败原因标签 | 趋势分析、Flaky Test 识别 |

### 4.2 采集方式

| CI 平台 | 采集接口 | 说明 |
|---------|---------|------|
| Jenkins | REST API (`/job/{name}/{build}/consoleText`) | 通过 `jenkins` Skill 拉取 |
| GitHub Actions | Artifact API + Workflow Run Logs | 通过 `ci-monitor` / `cicd-pipeline` Skill 拉取 |
| GitLab CI | Job Artifacts + Trace API | 扩展采集模块实现 |

### 4.3 基线对比数据

| 维度 | 策略 |
|------|------|
| 最近成功构建 | 自动查找当前分支最近一次 `SUCCESS` 构建，获取其日志和环境信息 |
| diff 基线 | 以最近一次成功构建的 Commit 为 base，计算当前失败的变更范围 |
| 环境基线 | 对比成功构建与失败构建的环境快照，识别环境漂移 |

### 4.4 敏感信息脱敏

| 敏感类型 | 脱敏规则 | 脱敏时机 |
|---------|---------|---------|
| 密钥/Token | 匹配 `password`/`token`/`secret`/`api_key` 等关键字后的值 | 采集后、入 AI 前 |
| IP 地址 | 内网 IP 段（10.x/172.16-31.x/192.168.x）替换为 `[INTERNAL_IP]` | 采集后 |
| 内部域名 | 匹配公司域名后缀，替换为 `[INTERNAL_DOMAIN]` | 采集后 |
| 个人邮箱/用户名 | 替换为 `[USER]` | 采集后 |

### 4.5 历史失败模式库

| 维度 | 设计方案 |
|------|---------|
| 存储结构 | 错误特征向量（提取关键错误片段 + 上下文哈希） |
| 匹配算法 | 向量相似度（Embedding 余弦相似度）或特征哈希 |
| 命中后加速 | 历史匹配命中时，直接将历史修复方案注入 Prompt，缩短诊断时间 |
| 更新策略 | 每次新的自愈成功案例自动入库，定期去重和淘汰过期记录 |

### 4.6 存储与保留策略

| 数据类型 | 保留周期 | 存储介质 |
|---------|---------|---------|
| 原始构建日志 | 30 天 | 对象存储 / 本地磁盘 |
| 脱敏后日志 | 90 天 | 数据库 |
| 诊断报告 | 180 天 | 数据库 |
| Patch 文件 | 90 天 | 对象存储 |
| 审计日志 | 1 年 | 数据库 + 日志系统 |
| 历史失败模式 | 长期保留 | 向量数据库 |

## 第五章 · AI 分析与诊断引擎

### 5.1 模型选型

| 场景 | 推荐模型 | 说明 |
|------|---------|------|
| 默认诊断 | DeepSeek-V3 / GPT-4 / Claude-3 | 通用代码理解与生成 |
| 高复杂度问题 | GPT-4 / Claude-3-Opus | 上下文长、逻辑复杂时切换 |
| 本地部署 | Qwen-Coder / CodeLlama | 数据敏感场景，离线运行 |
| 微调模型 | 基于正负样本微调后的专用模型 | 长期运行后，用积累数据微调 |

### 5.2 Prompt 工程设计

| 组件 | 设计要点 |
|------|---------|
| 系统提示词 | 定义 AI 角色（CI/CD 自愈代理）、约束（不修改业务代码）、输出格式 |
| 上下文组装 | 按优先级组装：错误片段 > 构建日志 > DSL > diff > 历史故障 > 环境快照 |
| 输出格式约束 | 强制 JSON Schema 输出，包含：根因、错误类型、修复代码、置信度、说明 |
| 上下文截断 | 日志超长时保留尾部（错误附近）和头部（环境信息），中间摘要 |

### 5.3 分析输出规范

AI 诊断输出必须包含以下字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| `root_cause` | string | 根因描述，含文件路径和行号（如 `Jenkinsfile:42`） |
| `error_type` | enum | 编译 / 依赖 / 配置 / 测试 / 逻辑 / 环境 / 安全 / 不可自愈 |
| `fix_suggestion` | string | 修复后的代码片段或配置 |
| `confidence` | float | 置信度评分，0.0 ~ 1.0 |
| `explanation` | string | 诊断 reasoning，供人工 Review 参考 |
| `cannot_fix_reason` | string | 如不可自愈，说明原因 |

### 5.4 置信度评分机制

| 评分维度 | 权重 | 说明 |
|---------|------|------|
| 错误信息完整性 | 30% | 日志中错误信息是否清晰、完整 |
| 历史匹配度 | 25% | 是否与历史成功修复案例相似 |
| 修复方案可行性 | 25% | 修复代码是否语法正确、是否符合项目规范 |
| 上下文充分性 | 20% | 输入信息（日志+DSL+diff）是否足够支撑诊断 |

**阈值设定**：
- `≥ 0.85`：高置信，可直接自动修复
- `0.60 ~ 0.85`：中置信，生成修复但需人工确认
- `< 0.60`：低置信，降级为仅诊断报告

### 5.5 修复类型分类器

| 错误类型 | 判定规则 | 自愈策略 |
|---------|---------|---------|
| 编译错误 | 日志含 `Compilation failure` / `Build failed` / `syntax error` | 🟢 自动修复 |
| 依赖冲突 | 日志含 `Dependency resolution failed` / `version conflict` | 🟢 自动修复 |
| 配置语法 | 日志含 `Invalid configuration` / `YAML parsing error` | 🟢 自动修复 |
| 环境变量缺失 | 日志含 `env: not set` / `variable not found` | 🟢 自动修复 |
| 测试失败 | 日志含 `Tests failed` / `AssertionError` | 🟡 Patch 人工确认 |
| 逻辑变更 | diff 指向业务代码变更导致测试失败 | 🔴 不可自愈 |
| 安全问题 | 安全扫描失败 | 🔴 不可自愈 |

### 5.6 低置信度降级

当置信度 `< 0.60` 时：
- 不生成自动修复代码
- 输出结构化诊断报告（根因分析 + 建议方向 + 相关日志片段）
- 通知维护者，供人工参考

### 5.7 多轮重试增强

| 重试轮次 | 增强策略 |
|---------|---------|
| 第 1 轮 | 标准 Prompt，基础上下文 |
| 第 2 轮 | 携带第 1 轮失败原因（Reward 信号），扩大上下文窗口 |
| 第 3 轮 | 切换更强模型（如从 DeepSeek 切换到 GPT-4），注入更多历史案例 |
| 第 4~5 轮 | 进一步扩大上下文，尝试不同 Prompt 策略，携带全部历史失败链 |

**Reward 信号格式**：包含前序轮次的诊断结果、应用后的构建结果、失败原因摘要。

## 第六章 · 修复策略决策矩阵

### 6.1 决策矩阵总表

| 错误类型 | 有 Git + 有权限 | 有 Git + 无权限 | 无 Git |
|---------|----------------|----------------|--------|
| 编译 / 依赖 / 配置 | 开 fix 分支 → Commit → Push → 触发重建 → 提 PR | 生成 Patch 文件 → 通知维护者 | 生成 Patch 文件 → 通知运维 |
| 测试失败 / 逻辑变更 | 生成 Patch → 通知维护者（不自动重建） | 生成 Patch → 通知运维 | 生成诊断报告 → 通知运维 |
| 环境 / 基础设施 | 生成修复建议 → 通知运维 | 生成修复建议 → 通知运维 | 生成修复建议 → 通知运维 |
| CANNOT_FIX_SRC | 终止 → 通知人工 | 终止 → 通知人工 | 终止 → 通知人工 |
| 置信度 < 0.60 | 降级为诊断报告 | 降级为诊断报告 | 降级为诊断报告 |

### 6.2 自动修复路径

```
AI 生成修复
    │
    ▼
创建 fix 分支（fix/ci-selfheal-{job}-{build}）
    │
    ▼
应用修复到分支（Commit）
    │
    ▼
Push 到远程仓库
    │
    ▼
触发 CI 重建（fix 分支）
    │
    ▼
重建成功 → 自动提 PR（fix → target）
```

### 6.3 Patch 生成路径

| 维度 | 设计方案 |
|------|---------|
| Patch 格式 | Unified Diff 标准格式，兼容 `git apply` |
| 生成逻辑 | 对比原始 DSL 与 AI 修复后的 DSL，生成标准 diff |
| 打包内容 | Patch 文件 + 诊断报告（Markdown）+ 应用说明 |
| 输出位置 | `patches/{job_name}_fix_{timestamp}.diff` |

### 6.4 诊断报告模板

报告结构（Markdown）：

```markdown
# CI Self-Heal 诊断报告

## 摘要
- Job: {job_name}
- Build: #{build_number}
- 诊断时间: {timestamp}
- 置信度: {confidence}

## 根因分析
{root_cause}

## 修复建议
{fix_suggestion}

## 证据日志
{relevant_log_snippets}

## 推荐操作
{recommended_action}
```

**角色定制**：
- 开发人员版：侧重代码修复细节、变更 diff、测试建议
- 运维版：侧重环境信息、基础设施状态、配置变更建议

### 6.5 权限判断逻辑

| 检查项 | 方式 |
|--------|------|
| Git 仓库检测 | 检查当前目录是否为 Git 仓库，`git remote` 是否可获取 |
| 写权限判断 | `git push --dry-run` 测试，或检测 API Token/SSH Key 权限 |
| Token 管理 | Git Token、CI API Token 统一由 Secrets Manager 管理，不硬编码 |

### 6.6 通知路由规则

| 场景 | 通知对象 | 渠道 |
|------|---------|------|
| 有 Git 权限，自动修复成功 | 维护者（Commit Author / Code Owner） | IM + 邮件 |
| 有 Git 权限，需人工确认 | 维护者 | IM + 邮件 |
| 无 Git 权限 / 无 Git | 运维团队 | IM |
| 安全相关 | 安全团队 | IM + 邮件 + On-Call |
| 熔断触发 | 值班运维 + 团队负责人 | 电话/短信 + IM |

## 第七章 · 验证与重建闭环

### 7.1 CI 触发机制

| 场景 | 触发方式 |
|------|---------|
| 自动修复 Push 后 | 由 Git 仓库 Webhook 自动触发 CI（标准流程） |
| 手动应用 Patch 后 | 通过 `jenkins` / `cicd-pipeline` Skill API 手动触发 |
| 验证构建 | 指定参数构建，标记为 `self-heal-verify` 类型 |

### 7.2 验证标准

| 检查项 | 通过标准 |
|--------|---------|
| 原有失败项 | 之前失败的 Stage/Job 全部通过 |
| 无新回归 | 未引入新的失败或警告 |
| 覆盖率不下降 | 代码覆盖率 ≥ 上一次成功构建 |
| 构建产物完整性 | 产物清单与基线一致 |

### 7.3 验证结果处理

| 结果 | 处理动作 |
|------|---------|
| ✅ 成功 | 自动提 PR，附带 AI 诊断说明，等待人工 Review |
| ❌ 失败 | 进入重试判断（当前重试次数 +1），携带 Reward 信号重新诊断 |
| ⏱️ 超时 | 视为失败，进入重试逻辑 |

### 7.4 PR 自动创建

| 维度 | 设计 |
|------|------|
| PR 标题模板 | `[ci-selfheal] Auto-fix: {job_name} #{build_number} — {error_type}` |
| PR 描述模板 | 含诊断摘要、根因、修复说明、验证结果、检查清单 |
| 标签 | `auto-fix` / `ci-selfheal` / `{error_type}` |
| Reviewer 指派 | 自动指派 Code Owner 或最近修改相关文件的开发者 |

### 7.5 人工 Review 指引

为 Reviewer 提供快速判断辅助信息：
- 变更 diff 高亮
- AI 诊断 reasoning
- 验证构建结果链接
- 一键接受 / 拒绝 / 修改按钮
- 检查清单（是否涉及业务代码、是否修改安全配置、测试是否通过）

## 第八章 · 重试与熔断机制

### 8.1 重试策略

| 维度 | 设计 |
|------|------|
| 最大重试次数 | 5 次（可配置） |
| 退避策略 | 指数退避：第 1 次立即，第 2 次 5 分钟，第 3 次 15 分钟，第 4 次 30 分钟，第 5 次 60 分钟 |
| 重试条件 | 仅自动修复路径失败时重试；Patch/报告路径不重试 |

### 8.2 增强重试

| 增强维度 | 策略 |
|---------|------|
| Reward 信号 | 每次重试携带前序失败的诊断结果、应用后的错误日志、失败原因摘要 |
| 扩大上下文 | 逐步增加历史构建日志、更多环境信息、相关文件内容 |
| 模型切换 | 第 3 轮后自动切换到更强模型（如 GPT-4 / Claude-3-Opus） |

### 8.3 熔断触发条件

| 触发条件 | 阈值 |
|---------|------|
| 单 Job 连续失败 | 同一 Job 连续 5 次重试全部失败 |
| 全局熔断 | 时间窗口（如 1 小时）内自愈失败率超过 80% |
| 异常模式 | 连续出现同一不可自愈类型（如基础设施故障） |

### 8.4 熔断行为

| 动作 | 说明 |
|------|------|
| 停止自动修复 | 新事件进入仅诊断模式，不生成修复 |
| 生成完整诊断报告 | 为每个失败构建输出详细诊断，供人工参考 |
| 升级通知 | 立即通知值班运维和团队负责人 |

### 8.5 熔断恢复

| 恢复方式 | 说明 |
|---------|------|
| 手动解除 | 运维人员确认问题根因已解决后，手动重置熔断状态 |
| 自动解除 | 熔断时间窗口（如 2 小时）过期后自动解除，恢复后状态重置 |
| 渐进恢复 | 解除后先对少量事件试点，确认正常后全面恢复 |

### 8.6 分级告警

| 级别 | 触发条件 | 通知渠道 |
|------|---------|---------|
| Info | 自愈成功、Patch 生成 | IM 机器人 |
| Warn | 置信度低降级、无 Git 权限、重试中 | IM + 邮件 |
| Critical | 熔断触发、连续失败超限、安全相关 | 电话/短信 + IM + On-Call |

### 9.1 PR 结果采集

### 9.4 Prompt 优化

| 维度 | 设计 |
|------|------|

| 优化周期 | 每两周分析一次 Prompt 效果，迭代优化 |
| 效果对比 | 对比指标：自愈成功率、平均置信度、人工接受率 |

### 9.5 历史失败模式库更新

| 维度 | 设计 |
|------|------|
| 新模式入库 | 每次新的自愈成功案例自动提取特征向量入库 |
| 去重策略 | 新入库前与已有模式计算相似度，高于阈值则合并更新 |

### 9.6 自愈效果报告

| 报告类型 | 周期 | 内容 |
