### 1.3 关键设计决策

#### 决策 1: 不单独部署 PostgreSQL/Redis

**原因**:
- GitLab CE Omnibus 包内置了 PostgreSQL 和 Redis
- 单独部署会增加运维复杂度
- 自愈功能本身不需要这些服务

**实现**:
- GitLab 配置中不设置 `postgresql['enable'] = false`
- GitLab 配置中不设置 `redis['enable'] = false`
- 使用 GitLab 内置的数据库和缓存

#### 决策 2: 不使用独立 Bridge 服务

**原因**:
- Bridge 服务的功能可以整合到 OpenClaw Skill 中
- 减少服务数量，降低运维复杂度
- Skill 与 AI 引擎无缝集成

**实现**:
- 创建 `ci-selfheal/` 目录存放自愈 Skill
- 将原有 Bridge 代码重构为 Skill 可调用的形式
- 提供 Webhook Listener 用于 Jenkins 集成

#### 决策 3: 所有端口绑定 127.0.0.1

**原因**:
- 安全考虑：不直接暴露到公网
- 本地开发/测试环境使用
- 生产环境通过 Nginx HTTPS 反向代理暴露

**实现**:
- 所有 `--publish` 参数使用 `127.0.0.1:HOST_PORT:CONTAINER_PORT`
- Nginx 监听宿主机端口，通过容器名转发到后端

#### 决策 4: Nginx 统一 HTTPS 入口

**原因**:
- Jenkins/GitLab/OpenClaw 各服务需要统一 HTTPS 访问
- 自签名证书管理集中在 Nginx，后端服务使用 HTTP
- 避免每个服务单独配置 SSL 的复杂性

**实现**:
- Nginx 监听 18440-18446 端口范围，映射到容器内 8440-8446
- 使用 `devopsclaw.crt` 统一证书
- `ensure_nginx_proxy()` 函数自动检测后端服务并生成对应 conf

## 三、Docker Compose 配置（可选）

> **注意：** `deploy_all.sh` 默认使用 `docker run` 逐个启动容器。`docker-compose.yml` 作为备用编排方案，参数通过环境变量 `.env` 控制。

### 3.1 完整配置

```yaml
version: '3.8'

networks:
  devopsclaw-network:
    driver: bridge
    name: ${DOCKER_NETWORK:-devopsclaw-network}

volumes:
  jenkins-home:
    name: ${COMPOSE_PROJECT_NAME:-devopsclaw}_jenkins-home
    driver: local
  openclaw-data:
    name: ${COMPOSE_PROJECT_NAME:-devopsclaw}_openclaw-data
    driver: local
  gitlab-config:
    name: ${COMPOSE_PROJECT_NAME:-devopsclaw}_gitlab-config
    driver: local
  gitlab-logs:
    name: ${COMPOSE_PROJECT_NAME:-devopsclaw}_gitlab-logs
    driver: local
  gitlab-data:
    name: ${COMPOSE_PROJECT_NAME:-devopsclaw}_gitlab-data
    driver: local
  nginx-logs:
    name: ${COMPOSE_PROJECT_NAME:-devopsclaw}_nginx-logs
    driver: local

services:
  openclaw:
    image: ${OPENCLAW_IMAGE:-ghcr.io/openclaw/openclaw:latest}
    container_name: ${OPENCLAW_CONTAINER_NAME:-devopsclaw-openclaw}
    restart: unless-stopped
    user: "1000:1000"
    cap_drop:
      - ALL
    security_opt:
      - no-new-privileges:true
    read_only: true
    tmpfs:
      - /tmp:rw,noexec,nosuid,size=64m
    networks:
      - devopsclaw-network
    ports:
      - "${OPENCLAW_BIND:-127.0.0.1}:${OPENCLAW_PORT:-18789}:18789"
    volumes:
      - openclaw-data:/home/node/.openclaw
    environment:
      - OPENCLAW_GATEWAY_TOKEN=${OPENCLAW_GATEWAY_TOKEN}
      - LOG_LEVEL=${LOG_LEVEL:-INFO}
    command: >
      node openclaw.mjs gateway --allow-unconfigured

  jenkins:
    image: ${JENKINS_IMAGE:-jenkins/jenkins:lts-jdk21}
    container_name: ${JENKINS_CONTAINER_NAME:-devopsclaw-jenkins}
    restart: unless-stopped
    networks:
      - devopsclaw-network
    ports:
      - "${JENKINS_BIND:-127.0.0.1}:${JENKINS_PORT_WEB:-18081}:8080"
      - "${JENKINS_BIND:-127.0.0.1}:${JENKINS_PORT_AGENT:-50000}:50000"
    volumes:
      - jenkins-home:/var/jenkins_home
      - /var/run/docker.sock:/var/run/docker.sock
      - ./jjb-configs:/var/jenkins_home/jjb-configs:ro
    environment:
      - JENKINS_OPTS=--prefix=/jenkins
      - JAVA_OPTS=${JENKINS_JAVA_OPTS:--Xmx2g -Xms512m}
      - JENKINS_URL=${JENKINS_URL:-http://127.0.0.1:18081/jenkins}
      - OPENCLAW_HOST=${OPENCLAW_CONTAINER_NAME:-devopsclaw-openclaw}
      - OPENCLAW_PORT=18789
      - DEFAULT_MODEL=${DEFAULT_MODEL:-deepseek-reasoner}
      - MAX_RETRY=${MAX_RETRY:-5}
      - LOG_LEVEL=${LOG_LEVEL:-INFO}
    depends_on:
      - openclaw

  gitlab:
    image: ${GITLAB_IMAGE:-gitlab/gitlab-ce:latest}
    container_name: ${GITLAB_CONTAINER_NAME:-devopsclaw-gitlab}
    restart: unless-stopped
    hostname: ${GITLAB_HOSTNAME:-gitlab.devopsclaw.local}
    networks:
      - devopsclaw-network
    ports:
      - "${GITLAB_BIND:-127.0.0.1}:${GITLAB_PORT_HTTP:-19092}:80"
      - "${GITLAB_BIND:-127.0.0.1}:${GITLAB_PORT_HTTPS:-19443}:443"
      - "${GITLAB_BIND:-127.0.0.1}:${GITLAB_PORT_SSH:-2222}:22"
    volumes:
      - gitlab-config:/etc/gitlab
      - gitlab-logs:/var/log/gitlab
      - gitlab-data:/var/opt/gitlab
    shm_size: ${GITLAB_SHM_SIZE:-512m}
    environment:
      GITLAB_OMNIBUS_CONFIG: |
        external_url 'http://${GITLAB_HOSTNAME:-gitlab.devopsclaw.local}';
        gitlab_rails['gitlab_shell_ssh_port'] = ${GITLAB_PORT_SSH:-2222};
        nginx['listen_port'] = 80;
        nginx['listen_https'] = false;

  nginx:
    image: ${NGINX_IMAGE:-nginx:alpine}
    container_name: ${NGINX_CONTAINER_NAME:-devopsclaw-nginx}
    restart: always
    networks:
      - devopsclaw-network
    ports:
      - "${NGINX_BIND:-127.0.0.1}:443:443"
      - "${NGINX_BIND:-127.0.0.1}:${NGINX_PORT_JENKINS:-18440}:8440"
      - "${NGINX_BIND:-127.0.0.1}:${NGINX_PORT_GITLAB:-18441}:8441"
      - "${NGINX_BIND:-127.0.0.1}:${NGINX_PORT_OPENCLAW:-18442}:8442"
    volumes:
      - ./deploy_nginx/nginx/nginx.conf:/etc/nginx/nginx.conf:ro
      - ./deploy_nginx/nginx/conf.d:/etc/nginx/conf.d:ro
      - ./deploy_nginx/nginx/ssl:/etc/nginx/ssl:ro
      - nginx-logs:/var/log/nginx
    depends_on:
      - jenkins
      - gitlab
      - openclaw
```

### 3.2 配置详解

#### 3.2.1 Network 配置

```yaml
networks:
  devopsclaw-network:
    driver: bridge
    name: ${DOCKER_NETWORK:-devopsclaw-network}
```

**说明**:
- 创建独立的 bridge 网络，网络名可通过 `DOCKER_NETWORK` 环境变量自定义
- 所有服务加入此网络，通过容器名互相访问
- 容器名解析: `devopsclaw-jenkins`, `devopsclaw-openclaw`, `devopsclaw-gitlab`, `devopsclaw-nginx`

#### 3.2.2 Volume 配置

| Volume 名 | 挂载路径 (容器内) | 用途 |
|----------|------------------|------|
| `jenkins-home` | `/var/jenkins_home` | Jenkins 数据、配置、插件、Job 历史 |
| `openclaw-data` | `/home/node/.openclaw` | OpenClaw 配置、模型缓存 |
| `gitlab-config` | `/etc/gitlab` | GitLab 配置文件 |
| `gitlab-logs` | `/var/log/gitlab` | GitLab 日志 |
| `gitlab-data` | `/var/opt/gitlab` | GitLab 数据（仓库、数据库等） |
| `nginx-logs` | `/var/log/nginx` | Nginx 访问及错误日志 |
| `mantisbt-web` | `/var/www/html` | MantisBT Web 文件 |
| `mantisbt-db-data` | `/var/lib/mysql` | MantisBT MariaDB 数据 |

所有卷名使用 `${COMPOSE_PROJECT_NAME:-devopsclaw}_` 前缀，支持多项目部署隔离。

#### 3.2.3 端口参数化

所有端口通过环境变量控制，格式为 `${VAR:-默认值}`:

| 变量 | 默认值 | 说明 |
|------|--------|------|
| `JENKINS_PORT_WEB` | 18081 | Jenkins Web 端口 |
| `JENKINS_PORT_AGENT` | 50000 | Jenkins Agent 端口 |
| `OPENCLAW_PORT` | 18789 | OpenClaw Gateway 端口 |
| `GITLAB_PORT_HTTP` | 19092 | GitLab HTTP 端口 |
| `GITLAB_PORT_HTTPS` | 19443 | GitLab HTTPS 端口 |
| `GITLAB_PORT_SSH` | 2222 | GitLab SSH 端口 |
| `NGINX_PORT_JENKINS` | 18440 | Nginx → Jenkins |
| `NGINX_PORT_GITLAB` | 18441 | Nginx → GitLab |
| `NGINX_PORT_OPENCLAW` | 18442 | Nginx → OpenClaw |
| `NGINX_PORT_MANTISBT` | 18443 | Nginx → MantisBT |

#### 3.2.4 OpenClaw 安全配置

```yaml
user: "1000:1000"                    # 非 root 用户运行
cap_drop:
  - ALL                              # 丢弃所有 Linux capabilities
security_opt:
  - no-new-privileges:true          # 禁止提权
read_only: true                       # 只读文件系统
tmpfs:
  - /tmp:rw,noexec,nosuid,size=64m  # 临时可写目录
ports:
  - "${OPENCLAW_BIND:-127.0.0.1}:${OPENCLAW_PORT:-18789}:18789"  # 参数化端口
```

**安全设计原则**:
1. **最小权限原则**: 使用非 root 用户，丢弃所有 capabilities
2. **只读文件系统**: 防止容器被篡改
3. **临时目录限制**: `/tmp` 目录限制大小，禁止 setuid
4. **网络隔离**: 仅绑定 127.0.0.1，不暴露到公网

#### 3.2.5 Jenkins 配置

**关键设计**:
- **JDK 21**: 使用 `jenkins/jenkins:lts-jdk21` 镜像，与部署脚本一致
- **无 `user: root`**: Jenkins 容器以默认 jenkins 用户(UID 1000)运行
- **CSRF 禁用**: `deploy_jenkins.sh` 启动时会注入 `-Dhudson.security.csrf.GlobalCrumbIssuerConfiguration.DISABLE_CSRF_PROTECTION=true`

**关键挂载**:
```yaml
volumes:
  - jenkins-home:/var/jenkins_home
  - /var/run/docker.sock:/var/run/docker.sock    # Docker-in-Docker
  - ./jjb-configs:/var/jenkins_home/jjb-configs:ro  # JJB 配置（只读）
```

#### 3.2.6 GitLab 配置

**关键设计**:
- **不单独部署 PostgreSQL/Redis**: 使用 GitLab 内置的服务
- **shm_size**: 分配 `${GITLAB_SHM_SIZE:-512m}` 共享内存
- **WSL/Windows 兼容**: 支持 `GITLAB_USE_NAMED_VOLUMES=true` 使用 Docker 命名卷

#### 3.2.7 Nginx 配置

**关键设计**:
- **HTTPS 统一入口**: 监听 8440-8446 端口，映射到宿主机 18440-18446
- **自动检测**: `ensure_nginx_proxy()` 函数自动检测运行中的后端容器并生成对应的 conf 文件
- **SSL 证书**: 使用 `deploy_nginx/nginx/ssl/devopsclaw.crt` 统一自签名证书

## 九、故障排查

### 9.1 常见问题

#### 问题 1: 容器启动失败

**症状**: 容器无法启动

**排查步骤**:

```bash
# 查看容器状态
docker ps -a --filter "name=devopsclaw"

# 查看容器日志
docker logs devopsclaw-<service-name>

# 查看详细日志（最后 100 行）
docker logs --tail 100 devopsclaw-<service-name>
```

**常见原因**:

| 错误信息 | 原因 | 解决方案 |
|---------|------|---------|
| `port is already allocated` | 端口被占用 | 检查并释放端口，或修改 `.env` 中的端口配置 |
| `pull access denied` | 镜像拉取失败 | 检查网络连接，配置镜像加速 |
| `no space left on device` | 磁盘空间不足 | 清理 Docker 镜像/容器，扩容磁盘 |
| `permission denied` | 权限问题 | 使用 sudo 运行，检查目录权限 |
| `device signature expired` | OpenClaw 时间偏移 | 运行 `sudo ./deploy_all.sh --reset-openclaw-device` |

#### 问题 2: Jenkins 无法访问

**症状**: 浏览器无法打开 https://127.0.0.1:18440/jenkins

**排查步骤**:

```bash
# 检查 Jenkins 容器状态
docker ps --filter "name=devopsclaw-jenkins"

# 检查容器日志
docker logs devopsclaw-jenkins

# 检查 Nginx 状态
docker ps --filter "name=devopsclaw-nginx"
docker logs devopsclaw-nginx
```

**可能原因**:

1. **Jenkins 正在初始化**: 首次启动需 2-3 分钟，查看日志等 "Jenkins is fully up and running"
2. **Nginx 未启动**: `sudo ./deploy_all.sh --deploy-nginx-standalone`
3. **端口被占用**: `netstat -tlnp | grep 18440`

#### 问题 3: OpenClaw 健康检查失败

## 十、Nginx 反向代理设计（v5.0.0 更新）

> **注意:** 此节描述 Nginx 的角色和设计原则。实际端口映射为 18440-18446 范围，详见 [二、端口分配](#二端口分配)。

### 10.1 定位与职责

Nginx 是整个 CI 平台的唯一 HTTPS 流量入口，负责：

| 职责 | 说明 |
|------|------|
| **SSL 终结** | 统一处理 HTTPS 加密，后端服务用 HTTP 明文通信 |
| **反向代理** | 根据端口将请求转发到对应的后端容器 |
| **访问日志** | 所有 Web 请求统一记录 |
| **安全隔离** | 后端 Web 服务不暴露宿主机端口，仅通过 Docker 内部网络可达 |

### 10.2 端口转发映射表

| 用户访问地址 | Nginx 监听端口 | 转发到后端 | 说明 |
|-------------|---------------|-----------|------|
| `https://127.0.0.1:18440/jenkins` | 8440 | `http://devopsclaw-jenkins:8080/jenkins` | Jenkins |
| `https://127.0.0.1:18441` | 8441 | `http://devopsclaw-gitlab:80` | GitLab |
| `https://127.0.0.1:18442` | 8442 | `http://devopsclaw-openclaw:18789` | OpenClaw |
| `https://127.0.0.1:18443` | 8443 | `http://devopsclaw-mantisbt:80` | MantisBT |
| `https://127.0.0.1:18444` | 8444 | (可选) | Registry |
| `https://127.0.0.1:18445` | 8445 | (可选) | Harbor |
| `https://127.0.0.1:18446` | 8446 | (可选) | SonarQube |

### 10.3 核心设计原则

| 原则 | 说明 |
|------|------|
| **Nginx 是唯一暴露 HTTPS 端口的容器** | 后端服务不绑定宿主机端口，仅在 Docker 内部网络可达 |
| **通过容器名转发** | 走 Docker 内部 DNS（`devopsclaw-jenkins`），不走 127.0.0.1 |
| **按端口隔离** | 不同服务用不同端口，不用子路径，避免兼容性问题 |
| **SSL 统一管理** | 一套自签名证书（`devopsclaw.crt`），一处更新，全站生效 |
| **自动检测** | `ensure_nginx_proxy()` 自动检测运行中的容器并生成对应 conf |

### 10.4 方案优势

| 维度 | 说明 |
|------|------|
| **SSL 统一管理** | 一套证书，一处更新，无需在每个服务配置 |
| **统一审计日志** | 所有 Web 访问记录在一个文件，便于审计 |
| **后端不暴露公网** | 后端服务仅通过 Docker 内部网络可达，更安全 |
| **无兼容性坑** | 不用子路径、不用域名，每个服务独立端口 |
| **各数据库独立** | GitLab/Harbor 用内置数据库，互不影响 |
| **易于扩展** | 新增服务只需添加 Nginx 端口配置 |
| **灵活配置** | 可配置负载均衡、限流、缓存等高级功能 |

### 10.7 风险与应对

| 风险 | 应对措施 |
|------|---------|
| **Nginx 单点故障** | `restart: always` + 健康检查 + 监控告警 |
| **证书管理** | 使用环境变量或 Docker Secret 管理证书，定期轮换 |
| **配置复杂** | 提供配置模板和一键生成工具 |
| **性能开销** | Nginx 性能损耗很小，可忽略；高并发时可横向扩展 |
| **调试困难** | 统一日志输出，提供 access.log 和 error.log 分析工具 |
| **自签名证书不被信任** | 所有客户端安装 CA 证书或配置 `insecure-registries` |

### 10.8 为什么用端口隔离而不是子路径？

子路径方案（如 `https://IP/gitlab`、`https://IP/jenkins`）在实际落地时有严重的兼容性问题：

**问题 1: 绝对路径问题**
- GitLab、Jenkins、Harbor 等应用的页面链接、API 调用、WebSocket 连接会大量使用绝对路径
- 应用内部可能硬编码了 `/assets/`、`/api/` 等路径，无法识别子路径前缀

**问题 2: 配置复杂度**
- 需要修改每个应用的 `external_url`、`context path` 等配置
- 不同应用的配置方式不同：
  - Jenkins: `--prefix=/jenkins`
  - GitLab: `external_url 'http://example.com/gitlab'`
  - Harbor: 需要修改 Nginx 配置和多个组件配置

**问题 3: 功能限制**
- WebSocket 连接可能无法正常工作
- 某些插件/扩展不支持子路径
- API 文档、重定向链接可能出错

**问题 4: 排查困难**
- 问题排查极其困难，而且每个应用行为不同
- 需要同时检查 Nginx 配置和应用配置

**端口隔离是最简单、最稳定、零兼容性问题的方案**：
- 每个服务独立端口，互不干扰
- 应用内部使用默认配置，无需修改
- 任何支持标准 HTTP 的应用都可以无缝接入
- 问题排查清晰：Nginx 层或应用层

### 10.9 版本对比

| 特性 | v4.0.0（无 Nginx） | v4.1.0（带 Nginx） |
|------|-------------------|-------------------|
| 后端服务端口绑定 | 绑定 127.0.0.1 | 不绑定宿主机端口 |
| SSL 配置 | 每个服务单独配置 | Nginx 统一配置 |
| 访问日志 | 分散在各容器 | 统一在 Nginx |
| 安全等级 | 中等（本地绑定） | 高（后端不暴露） |
| 配置复杂度 | 低 | 中等（需配置 Nginx） |
| 扩展性 | 一般 | 好 |

## 十一、端口与 IP 智能扫描（v5.1.0 新增）

### 11.1 设计目标

部署前自动扫描宿主机环境，检测端口占用和 Docker 网络冲突，**自动分配未占用的端口**，避免人工排查。

### 11.2 当前问题

#### 问题 A: 端口冲突检测只是警告，不会自动解决

`check_ports()` 的逻辑：

```bash
# 当前逻辑: netstat 检测 → 打印警告 → 问用户是否继续
if netstat -tlnp | grep -q ":$port "; then
    log_warn "端口被占用"
    conflicts+=("$port")
fi
# 用户只能 y/n，没有自动选替代端口的能力
```

#### 问题 B: 没有"端口池"概念

所有端口散落在 6 个文件中，没有一个统一的、可编程读取的端口注册表。

#### 问题 C: 预留端口也被静态分配

18444/18445/18446 是为未部署服务预留的，但已在 `docker-compose.yml` 和 `deploy_nginx.sh` 中实际绑定，白白占用。

#### 问题 D: Docker 占用的端口检测不完整

`ss -tlnp` 看到的进程是 `docker-proxy`，无法区分"被本项目的另一个容器占用"和"被不相干的进程占用"。

### 11.3 端口扫描设计

#### 端口注册表（统一数据源）

```python
PORT_REGISTRY = {
    "jenkins":    {"web": 18081, "agent": 50000},
    "openclaw":   {"web": 18789, "agent": 8080},
    "gitlab":     {"http": 19092, "https": 19443, "ssh": 2222},
    "mantisbt":   {"web": 19093},
    "mariadb":    {"db": 3307},
    "nginx": {
        "jenkins": 18440, "gitlab": 18441, "openclaw": 18442,
        "mantisbt": 18443, "registry": 18444, "harbor": 18445,
        "sonarqube": 18446,
    },
    "webhook":    5000,
}
```

#### 扫描策略

```
step 1: ss -tlnp                     → 宿主机所有 LISTEN 端口
step 2: docker ps --format {{.Ports}} → Docker 容器已暴露端口
step 3: 合并去重                       → 完整已占用端口列表
step 4: 对每个"默认端口"找第一个可用偏移 → AUTO_PORT_MAP
step 5: 写入 .env.auto（不覆盖用户手动配置的 .env）
```

#### 端口偏移策略（按服务优先级）

| 优先级 | 端口范围 | 适用服务 | 偏移策略 |
|--------|---------|---------|---------|
| P0 | 10xxx 整千 | 18081/50000/19092 | +1,+2 依次尝试 |
| P1 | 18440-18449 | Nginx HTTPS | 在范围内跳跃 |
| P2 | 3307 | MariaDB | 3307→3308→3309 |
| P3 | 5000 | Webhook | 5000→5001→5002 |

### 11.4 Docker Bridge IP 扫描

#### 当前网络配置

```yaml
networks:
  devopsclaw-network:
    driver: bridge
    name: ${DOCKER_NETWORK:-devopsclaw-network}
```

没有显式配置 `subnet`，Docker 自动从默认池 `172.17.0.0/16` 起分配。

#### 冲突检测扫描

```python
def scan_docker_network():
    # 1. 列出所有 Docker 网络及其子网
    # docker network ls -q | xargs docker network inspect
    
    # 2. 检查宿主路由表是否与 Docker bridge 重叠
    # ip route | grep -E '172\.(1[6-9]|2[0-9]|3[01])\.'
    
    # 3. 检测 Hyper-V/WSL/VPN 网段冲突
    # ip addr show | grep -E '(172\.|10\.|192\.168\.)'
```

#### 冲突等级

| 等级 | 场景 | 影响 | 处理 |
|------|------|------|------|
| **高危** | Docker bridge 子网 ∈ 宿主非 Docker 路由 | 容器外部通信中断 | 必须修改 subnet |
| **中危** | Docker bridge 与 Hyper-V/WSL Switch 重叠 | 容器→宿主部分不通 | 建议修改 subnet |
| **低危** | 多个 Docker bridge 相邻不重叠 | 无影响 | 无需处理 |

### 11.5 执行顺序

```
check_root() → scan_environment() → check_docker() → setup_env() → select_mode() → deploy()
                   ↓产出
            ┌──────┴──────────────┐
            │ OCCUPIED_PORTS      │ → 内存变量
            │ AUTO_PORT_MAP       │ → .env.auto 文件
            │ NETWORK_CONFLICTS   │ → stdout 警告
            │ EXISTING_VOLUMES    │ → 卷冲突检查
            └─────────────────────┘
```

### 12.2 当前问题

#### 问题 A: docker-compose 与 docker-run 卷名不一致

```
docker-compose 卷:  devopsclaw_jenkins-home     (有 COMPOSE_PROJECT_NAME 前缀)
docker-run 卷:      jenkins-home                 (无前缀，deploy_jenkins.sh 默认)
```

#### 问题 B: 卷名冲突风险

同一 Docker 宿主机上，两个用户使用相同 `COMPOSE_PROJECT_NAME` 会导致数据卷互相覆盖。

#### 问题 C: 数据备份缺失

- 部署前不检查卷是否已存在数据（可能误清空）
- 重置脚本直接 `docker volume rm` 没有提示备份
- 没有跨宿主机迁移指引
