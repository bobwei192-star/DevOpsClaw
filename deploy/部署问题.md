# Jenkins standalone 会重建 Nginx，且 Nginx 地址文案错误写成 HTTP

## 问题背景

在 `DevOpsClaw_new/deploy` 目录执行：

```bash
sudo bash deploy_all.sh --deploy-jenkins-standalone
```

实际观察到：

1. Jenkins standalone 部署完成后，会继续执行 `Nginx 反向代理集成`
2. 集成阶段会先清理已有 `devopsclaw-nginx` 容器，再重新创建
3. Jenkins 摘要里给出的 Nginx 地址是：

```text
http://127.0.0.1:18440/jenkins/
```

但 Nginx 配置实际监听的是：

```nginx
listen 8440 ssl;
```

因此这里的协议文案与真实配置不一致，应为 `https`

## 复现日志关键片段

### 1. Jenkins standalone 会继续执行 Nginx 集成

```text
=== Jenkins 一键部署/修复 ===
...
[INFO] Jenkins 独立部署完成

=== Nginx 反向代理集成 ===
```

### 2. 已有 Nginx 容器会被停止并删除

```text
[INFO] 清理旧 Nginx 容器（如存在）...
devopsclaw-nginx
devopsclaw-nginx
```

### 3. Jenkins 摘要文案给的是 HTTP

```text
Jenkins 访问地址:
  - Nginx (推荐): http://127.0.0.1:18440/jenkins/
  - 直连: http://127.0.0.1:18081/jenkins/
```

### 4. 实际 Nginx 监听为 SSL

`deploy/deploy_nginx/nginx/conf.d/jenkins.conf`:

```nginx
server {
    listen 8440 ssl;
    listen [::]:8440 ssl;
```

`deploy/deploy_nginx/nginx/conf.d/gitlab.conf`:

```nginx
server {
    listen 8441 ssl;
    listen [::]:8441 ssl;
```

## 根因定位

## 1. Jenkins standalone 设计上会无条件调用 `ensure_nginx_proxy()`

`deploy/deploy_all.sh` 中：

```bash
deploy_jenkins_standalone() {
    ...
    "$JENKINS_DEPLOY_SCRIPT" --standalone
    ...
    ensure_nginx_proxy
}
```

也就是说，只要执行：

```bash
sudo bash deploy_all.sh --deploy-jenkins-standalone
```

脚本就不仅会部署 Jenkins，还会进入一套“自动检测后端服务并重建 Nginx 容器”的流程。

## 2. `ensure_nginx_proxy()` 内部会先删旧容器，再重新起一个新的

`deploy/deploy_all.sh` 中：

```bash
if docker ps -q --filter "name=$NGINX_CONTAINER_NAME" 2>/dev/null | grep -q .; then
    docker stop "$NGINX_CONTAINER_NAME" 2>/dev/null || true
fi
if docker ps -aq --filter "name=$NGINX_CONTAINER_NAME" 2>/dev/null | grep -q .; then
    docker rm -f "$NGINX_CONTAINER_NAME" 2>/dev/null || true
fi
```

所以当前行为不是“复用已有 Nginx 并增量刷新配置”，而是“删掉原有 Nginx 容器再重建”。

## 3. Jenkins / GitLab standalone 摘要文案仍然写的是 HTTP

`deploy/deploy_jenkins/deploy_jenkins.sh` 中：

```bash
elif [[ "$mode" == "with_nginx" ]]; then
    echo -e "  - Nginx (推荐): ${CYAN}http://127.0.0.1:18440/jenkins/${NC}"
```

`deploy/deploy_gitlab/deploy_gitlab.sh` 中：

```bash
echo -e "  - Nginx (推荐): ${CYAN}http://127.0.0.1:18441${NC}"
```

但实际 Nginx 配置是 SSL 监听，因此这两处文案都应改成 `https://...`

## 当前影响

## 1. 对 Jenkins standalone 的影响

- 会打断并重建已有 `devopsclaw-nginx`
- 如果原 Nginx 容器里还有其他运行时状态，当前实现不会保留
- 用户看到的访问地址文案会被误导为 HTTP

## 2. 对 GitLab standalone 的影响

有同类问题。

因为 `deploy_gitlab_standalone()` 也会在部署后调用：

```bash
ensure_nginx_proxy
```

所以单独部署 GitLab 时，同样会：

- 清理已有 Nginx 容器
- 自动检测 Jenkins / GitLab / OpenClaw
- 重建一个新的 `devopsclaw-nginx`

同时 GitLab standalone 摘要里的 Nginx 地址文案也错误写成了 `http://127.0.0.1:18441`

## 3. 对 OpenClaw standalone 的影响

同样会进入：

```bash
ensure_nginx_proxy
```

因此 OpenClaw standalone 也具有“重建 Nginx 容器”的同一类行为。

## 结论

当前 `deploy_all.sh` 里的 standalone 模式，本质上不是“只部署单个服务”，而是：

- 部署目标服务
- 然后执行一次共享的 Nginx 自动集成流程

所以：

1. **已有 Nginx 容器被清理掉，不是异常，是当前代码设计如此**
2. **Jenkins / GitLab standalone 的 Nginx 访问地址文案写成 HTTP，是代码错误**
3. **GitLab standalone 也有同样问题，不只 Jenkins**

## 建议修复

1. 将 Jenkins / GitLab standalone 输出中的 Nginx 地址统一改为 `https`
2. 将 `ensure_nginx_proxy()` 的行为从“删除重建容器”优化为：
   - 优先复用现有容器
   - 配置变更后 reload
   - 只有必要时才重建
3. 至少在日志里明确提示：
   - 将要清理并重建 Nginx 容器
   - 检测到哪些后端服务
   - 当前是全量重建，不是增量刷新

## 当前状态

- 已复现
- 已定位根因
- 已修复：
  - `ensure_nginx_proxy()` 改为优先复用现有 Nginx 容器并执行 `nginx -t && reload`
  - 仅当现有容器未运行、端口映射不满足、或挂载目录不是当前部署目录时才重建
  - 复用前增加 Docker 网络校验，现有 Nginx 容器必须挂在 `devopsclaw-network`
  - `nginx -t` 失败时直接输出真实错误，不再提示去看 `docker logs`
  - SSL 证书检查从“仅检查文件存在”升级为“校验证书和私钥是否匹配”
  - 发现 `.crt/.key` 缺失或不匹配时，会自动重生成对应自签名证书
  - 独立 `deploy_nginx.sh` 同步纳入 `devopsclaw.crt/key` 并做匹配校验
  - Jenkins / GitLab standalone 摘要中的 Nginx 推荐地址已改为 `https`

## 追加问题：外部浏览器无法访问 Nginx

后续排查确认 Jenkins 和 Nginx 链路本身正常：

- Jenkins 容器内访问 `http://127.0.0.1:8080/jenkins/` 返回 `403`
- 宿主机访问 `http://127.0.0.1:18081/jenkins/` 返回 `403`
- Nginx 容器访问 `http://devopsclaw-jenkins:8080/jenkins/` 返回 `403`
- 宿主机访问 `https://127.0.0.1:18440/jenkins/` 返回 `403`

`403` 在这里说明 Jenkins 已经响应，请求链路是通的。

真正导致用户自己电脑访问不了的原因是端口默认只绑定宿主机回环地址：

```text
127.0.0.1:18440->8440/tcp
127.0.0.1:18081->8080/tcp
```

这表示只能在服务器本机访问，不能通过 `https://<服务器IP>:18440/jenkins/` 从外部机器访问。

### 新增修复目标

给 `deploy_all.sh` 增加命令行选项：

```bash
--nginx-bind <IP>
```

用法示例：

```bash
sudo bash deploy_all.sh --deploy-jenkins-standalone --nginx-bind <服务器IP>
```

或：

```bash
sudo bash deploy_all.sh --deploy-jenkins-standalone --nginx-bind 0.0.0.0
```

要求：

- 该选项应覆盖 `NGINX_BIND`
- 适用于 Jenkins / GitLab / OpenClaw standalone 后续调用的 `ensure_nginx_proxy()`
- 当绑定地址变化时，现有 Nginx 容器端口映射不匹配，应触发安全重建
- 输出文案中的 Nginx 访问地址应显示用户传入的绑定地址

### 实现状态

- 已在 `deploy_all.sh` 增加全局选项 `--nginx-bind <IP>` / `--nginx-bind=<IP>`
- 该选项会导出 `NGINX_BIND`，可配合 Jenkins / GitLab / OpenClaw / Nginx standalone 使用
- Jenkins / GitLab / OpenClaw / Nginx 摘要输出已改为跟随 `NGINX_BIND`
- 当 `NGINX_BIND` 从 `127.0.0.1` 改为服务器 IP 或 `0.0.0.0` 时，`ensure_nginx_proxy()` 会因端口映射不匹配而重建 Nginx 容器

## 追加问题：GitLab 经 Nginx 访问 502 / 500

执行：

```bash
sudo bash deploy_all.sh --deploy-gitlab-standalone --nginx-bind 10.67.69.34
```

后访问：

```text
https://10.67.69.34:18441/
```

出现 `502 Bad Gateway`，随后排查发现：

```bash
docker exec devopsclaw-gitlab bash -lc 'curl -I http://127.0.0.1/-/health'
```

返回 `200 OK`，说明 GitLab 容器内部服务已经起来。

但从 Nginx 容器访问 GitLab 容器名：

```bash
docker exec devopsclaw-nginx sh -lc 'wget -S -O- http://devopsclaw-gitlab:80/-/health'
```

返回 `404 Not Found`。

访问：

```bash
docker exec devopsclaw-nginx sh -lc 'wget -S -O- http://devopsclaw-gitlab:80/users/sign_in'
```

返回 `500 Internal Server Error`。

### 初步判断

GitLab 容器内部服务是可用的，但当前 GitLab 部署仍按直接访问模式配置：

```bash
GITLAB_USE_HTTPS_PROXY=false
GITLAB_EXTERNAL_URL=http://127.0.0.1:19092
```

而外部实际访问入口是：

```text
https://10.67.69.34:18441
```

Nginx 反代时又把浏览器 Host 透传给 GitLab：

```nginx
proxy_set_header Host $http_host;
proxy_set_header X-Forwarded-Proto $scheme;
```

所以 GitLab 看到的 Host / Proto 与自身 `external_url` 不一致，容易导致路由、登录页、重定向和 Rails 页面异常。

### 后续修复方向

当执行 GitLab standalone 且传入 `--nginx-bind <服务器IP>` 时，应同步配置：

```bash
GITLAB_USE_HTTPS_PROXY=true
GITLAB_HOSTNAME=<服务器IP>
GITLAB_EXTERNAL_URL=https://<服务器IP>:18441
```

同时 `gitlab.conf` 应每次覆盖更新，确保反代头部与 HTTPS 入口一致，而不是只在文件不存在时创建。

### 实现状态

- 已在 `deploy_gitlab_standalone()` 中增加 HTTPS 反向代理模式自动配置：
  - 当 `NGINX_BIND != 127.0.0.1` 时，默认导出 `GITLAB_USE_HTTPS_PROXY=true`
  - 默认设置 `GITLAB_HOSTNAME=<NGINX_BIND>`
  - 默认设置 `GITLAB_EXTERNAL_URL=https://<GITLAB_HOSTNAME>:18441`
  - 若 `NGINX_BIND=0.0.0.0`，会尝试用 `hostname -I` 取首个服务器 IP，避免生成 `https://0.0.0.0:18441`
- `gitlab.conf` 已改为每次覆盖更新
- `gitlab.conf` 已补充：
  - `X-Forwarded-Ssl on`
  - `X-Forwarded-Port 18441`

## 最终解决总结

本轮问题已经收口，最终确认并修复了以下几类问题。

### 1. 子部署脚本存在但被误报为“不存在”

现象：

```text
Jenkins 部署脚本不存在: /home/jenkins/DevOpsClaw_new/deploy/deploy_jenkins/deploy_jenkins.sh
```

实际原因不是文件不存在，而是主脚本使用 `-x` 判断“可执行”，当子脚本缺少执行位时会进入错误分支。

处理方式：

- 给各 deploy 子脚本补执行权限
- 后续建议将错误文案从“不存在”改成“不存在或不可执行”，避免误导排障

### 2. Jenkins standalone 会重建 Nginx 容器

现象：

```text
=== Nginx 反向代理集成 ===
清理旧 Nginx 容器（如存在）...
```

根因：

`deploy_jenkins_standalone()` / `deploy_gitlab_standalone()` / `deploy_openclaw_standalone()` 都会调用共享的 `ensure_nginx_proxy()`，旧实现每次都会 `stop + rm + run`。

修复结果：

- `ensure_nginx_proxy()` 已改为优先复用现有 `devopsclaw-nginx`
- 仅当容器未运行、端口映射不匹配、挂载目录不匹配、Docker 网络不匹配时才重建
- 复用路径会执行 `nginx -t && nginx -s reload`

### 3. Nginx 推荐访问地址错误写成 HTTP

现象：

```text
Nginx (推荐): http://127.0.0.1:18440/jenkins/
```

实际 Nginx 配置监听的是 SSL：

```nginx
listen 8440 ssl;
```

修复结果：

- Jenkins / GitLab / OpenClaw / Nginx 摘要输出均改为 `https`
- 输出地址跟随 `NGINX_BIND`

### 4. SSL 证书和私钥不匹配

现象：

```text
SSL_CTX_use_PrivateKey("/etc/nginx/ssl/devopsclaw.key") failed
key values mismatch
```

根因：

旧逻辑只检查 `.crt` 和 `.key` 是否存在，不检查二者是否匹配。

修复结果：

- `deploy_all.sh` 和 `deploy_nginx.sh` 都增加证书/私钥匹配校验
- 发现缺失或不匹配时自动重生成自签名证书
- `nginx -t` 失败时直接打印真实错误，不再吞掉 stderr

### 5. 外部电脑无法访问 Nginx

现象：

服务器本机访问正常，但用户电脑无法访问：

```text
https://<服务器IP>:18440/jenkins/
```

根因：

默认端口只绑定在宿主机回环地址：

```text
127.0.0.1:18440->8440/tcp
```

修复结果：

`deploy_all.sh` 新增全局参数：

```bash
--nginx-bind <IP>
```

示例：

```bash
sudo bash deploy_all.sh --deploy-jenkins-standalone --nginx-bind 10.67.69.34
```

传入服务器 IP 后，Nginx 会绑定到该 IP，外部浏览器可访问：

```text
https://10.67.69.34:18440/jenkins/
```

### 6. GitLab 经 Nginx 访问 502 / 500

现象：

```text
https://10.67.69.34:18441/
502 Bad Gateway
```

进一步排查发现 GitLab 容器内部 `/-/health` 可返回 `200 OK`，但经 Nginx 容器访问时出现 `404 / 500`。

根因：

GitLab 仍按直接访问模式配置：

```text

```

但真实入口已经变成：

```text
https://10.67.69.34:18441
```

修复结果：

当 GitLab standalone 结合 `--nginx-bind <服务器IP>` 使用时，自动配置：

```bash

GITLAB_HOSTNAME=<服务器IP>
GITLAB_EXTERNAL_URL=https://<服务器IP>:18441
```

同时 `gitlab.conf` 改成每次覆盖更新，并补充：

```nginx
proxy_set_header X-Forwarded-Ssl on;
proxy_set_header X-Forwarded-Port 18441;
```

### 追加问题：GitLab standalone 未带 `--nginx-bind` 导致 Nginx 外部访问回退

复测时执行：

```bash
sudo ./deploy_all.sh --deploy-gitlab-standalone
```

由于未传 `--nginx-bind 10.67.69.34`，脚本按默认值 `127.0.0.1` 重新集成 Nginx：

```text
Nginx 监听地址: 127.0.0.1
127.0.0.1:18440->8440
127.0.0.1:18441->8441
127.0.0.1:18442->8442
```

结果：

- 外部浏览器无法访问 Jenkins：`https://10.67.69.34:18440/jenkins`
- 外部浏览器无法访问 OpenClaw：`https://10.67.69.34:18442`
- GitLab Nginx 地址也只输出 `https://127.0.0.1:18441`

根因：

- standalone GitLab 部署结束后会调用共享 `ensure_nginx_proxy()`
- `NGINX_BIND` 未显式设置时默认是 `127.0.0.1`
- 现有 Nginx 容器端口不匹配当前默认绑定后，会被判定“无法安全复用”并重建
- 因此单独修复 GitLab 误伤了已有 Jenkins/OpenClaw 的外部入口

已修复：

- `deploy_all.sh` 增加 `infer_existing_nginx_bind()` / `ensure_nginx_bind_default()`
- 未显式传 `--nginx-bind` 时，会优先沿用现有 Nginx 容器的监听地址
- `deploy_gitlab_standalone()` 会在部署 GitLab 前先解析 Nginx bind，避免 GitLab `external_url` 回退到 `127.0.0.1`
- 帮助文档补充 `--deploy-gitlab-standalone --nginx-bind <服务器IP>` 和 `--deploy-nginx-standalone --nginx-bind <服务器IP>` 示例

当前已被回退的环境需要手动恢复一次：

```bash
sudo ./deploy_all.sh --deploy-gitlab-standalone --nginx-bind 10.67.69.34
```

如果只想先恢复 Jenkins/OpenClaw 外部入口，不重建 GitLab，可先执行：

```bash
sudo ./deploy_all.sh --deploy-nginx-standalone --nginx-bind 10.67.69.34
```

## 追加问题：Jenkins Git Clone 自签名证书验证失败

### 问题现象

Jenkins Pipeline 执行 Git checkout 时失败：

```text
fatal: unable to access 'https://10.67.69.34:18441/ci/gitlab_repo_example.git/':
server certificate verification failed. CAfile: none CRLfile: none
```

### 根因分析

这个问题和部署脚本有关，但不是 GitLab 服务本身损坏。

当前链路是：

1. GitLab 通过 Nginx 暴露：

```text
https://10.67.69.34:18441
```

2. Nginx 使用部署脚本生成的自签名证书。
3. Jenkins Pipeline 中的 `git clone` 访问：

```text
https://10.67.69.34:18441/ci/gitlab_repo_example.git
```

4. Jenkins 容器里的 `git` 默认会校验证书链。
5. 自签名证书没有被 Jenkins 容器信任，因此 Git 报证书校验失败。

所以结论是：

- **和部署脚本有关**：因为部署脚本生成并使用了自签名 Nginx HTTPS 证书
- **不是 GitLab 仓库路径问题**
- **不是 Nginx 配置语法问题**
- **不是 GitLab 服务未启动问题**

### 临时处理方式

联调阶段可以在 Jenkinsfile 中对 checkout 包一层：

```groovy
withEnv(['GIT_SSL_NO_VERIFY=true']) {
    git branch: 'main',
        url: 'https://oauth2:<token>@10.67.69.34:18441/ci/gitlab_repo_example.git'
}
```

如果仍然报错，说明 Jenkins Job 实际执行的 Jenkinsfile 不是本地已修改版本，或者 Jenkins `git` 步骤没有吃到该环境变量。

### 长期处理方式

更稳妥的方案是把 Nginx 自签名证书或内部 CA 导入 Jenkins 容器信任链。

可选方案：

1. 临时联调：`GIT_SSL_NO_VERIFY=true`
2. 稳定内网：把 Nginx 自签名 CA/证书安装到 Jenkins 容器系统信任链
3. 正式环境：替换为公司 CA 或真实可信证书

### 当前建议

当前阶段优先用 `GIT_SSL_NO_VERIFY=true` 跑通闭环；后续如果这套环境要长期使用，应改为导入 CA，而不是长期关闭 Git SSL 校验。

## 追加问题：验收脚本仍引用旧 GitLab 示例仓库

### 问题现象

`verify-deployment_without_agent.sh` 的 GitLab skill 验证失败：

[gitlab-skill CLI 搜索项目] ... FAIL
got: ## No projects found
[gitlab-skill CLI 创建测试分支] ... FAIL
got: Resource not found
[GitLab API 搜索项目（获取 ID）] ... FAIL

### 根因分析

脚本仍按旧项目 `root/model_test` / `model_test` 做搜索、创建分支和 MR 创建测试；当前环境实际使用的 GitLab 示例仓库是：

- 未显式传 `--nginx-bind` 时，会优先沿用现有 Nginx 容器监听地址；无现有容器时才默认 `127.0.0.1`

# 综合根因分析与代码组织问题

## 一、代码组织问题：deploy_all.sh 与子脚本的边界不清

### 1.1 `ensure_nginx_proxy()` 放错了位置

**问题**：`ensure_nginx_proxy()` 是一个 ~260 行的 Nginx 专用函数，但定义在 [deploy_all.sh](file:///C:/Users/Tong/Desktop/DevOpsClaw/deploy/deploy_all.sh)（主控制脚本）中，而非 [deploy_nginx.sh](file:///C:/Users/Tong/Desktop/DevOpsClaw/deploy/deploy_nginx/deploy_nginx.sh)。

**后果**：形成循环依赖——

```
deploy_all.sh                                          deploy_nginx.sh
  ├── ensure_nginx_proxy() ← 在此定义                     └── standalone:
  ├── deploy_jenkins_standalone()                              source "$PROJECT_DIR/deploy_all.sh"
  │     └── ensure_nginx_proxy                                 ensure_nginx_proxy
  ├── deploy_gitlab_standalone()                         ← 子脚本竟要 source 主控制脚本
  │     └── ensure_nginx_proxy
  └── deploy_openclaw_standalone()
        └── ensure_nginx_proxy
```

`deploy_nginx.sh` 为了获取自己的核心功能，必须 `source` 主控制脚本，这是架构倒置。

**建议**：将 `ensure_nginx_proxy()` 移动到 `deploy_nginx.sh`，各 standalone 函数改为 source 子脚本。实际操作需注意 `set -e` 传递和变量作用域问题，建议分期重构。

### 1.2 standalone 函数职责不纯

`deploy_all.sh` 中每个 `--deploy-xxx-standalone` 函数的实际行为是 **"部署服务 + 强制集成 Nginx"**，而非"仅部署该服务"。例如：

```bash
deploy_jenkins_standalone() {
    "$JENKINS_DEPLOY_SCRIPT" --standalone     # 步骤1: 部署 Jenkins
    ensure_nginx_proxy                         # 步骤2: 重建/刷新 Nginx（不可跳过）
}
```

这导致单独部署 GitLab 时会误伤 Jenkins/OpenClaw 的 Nginx 外部入口（如果没传 `--nginx-bind`）。

### 1.3 端口配置分散在两处

`deploy_all.sh` 和 `deploy_nginx.sh` 各自维护了一份端口常量（`NGINX_PORT_JENKINS=18440` 等），不同步修改会引入隐性 bug。

## 三、全部问题的根因分类与共同规律

### 3.1 根因清单

| # | 问题 | 根因类别 | 一句话 |
|---|---|---|---|
| 1 | Jenkins standalone 重建 Nginx | 架构耦合 | standalone 函数强制调用 `ensure_nginx_proxy()` |
| 2 | Nginx 地址文案写成 HTTP | 假设不一致 | 代码写 HTTP，实际监听 SSL |
| 3 | SSL 证书/私钥不匹配 | 验证不足 | 只检查文件存在，不检查内容匹配 |
| 4 | 外部电脑无法访问 Nginx | 默认值不当 | `127.0.0.1` 绑定只能本机访问 |
| 5 | GitLab 经 Nginx 502/500 | 配置不联动 | `external_url` 与 Nginx 入口不一致 |
| 6 | GitLab standalone 回退 Nginx 绑定 | 静默副作用 | 未传 `--nginx-bind` 时重建容器造成降级 |
| 7 | Jenkins Git clone 证书验证失败 | 自签名证书 | 容器内不信任自签名证书 |
| 8 | 验收脚本引用旧仓库名 | 硬编码值 | 仓库路径写死，环境变化不感知 |
| 9 | 误用旧目录覆盖容器 | 运维流程 | 两个目录并存导致同步方向错误 |
| 10 | GitLab pre-receive 502 | GitLab 内部链路 | Rails Internal API 调用异常 |
| 11 | `ensure_nginx_proxy()` 放错文件 | 架构耦合 | Nginx 核心逻辑在控制脚本中 |
| 12 | deploy_nginx.sh source deploy_all.sh | 循环依赖 | 子脚本反 source 父脚本 |
| 13 | verify-deployment.sh 含真实 Token | 凭据泄露 | 硬编码 Token 作为默认值 |
| 14 | .global_settings.yaml 含 4 个真实凭据 | 凭据泄露 | 配置文件直接写 API Key / Token |

│                   │ 涉及问题: #1, #5, #6, #11, #12              │
├───────────────────┼─────────────────────────────────────────────┤
│ 模式 2: 默认值陷阱 │ 127.0.0.1 / HTTP / 文件存在即通过           │

│                   │ 涉及问题: #2, #3, #4, #7                     │
